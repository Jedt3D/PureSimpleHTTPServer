Advanced Tips & Tricks

Published 2026-03-05

Once you are comfortable serving files and writing rewrite rules, there are several techniques that can dramatically improve performance, developer experience, and operational resilience. This post covers the most impactful ones.

Running multiple instances for staging and production

Because PureSimpleHTTPServer is a single stateless binary with no shared process registry, you can run as many instances as you like side-by-side — each on its own port and serving its own root directory: 

# Production — port 8080, compressed production build
./PureSimpleHTTPServer --port 8080 --root ./dist --log ./logs/prod.log &

# Staging — port 8081, uncompressed staging build
./PureSimpleHTTPServer --port 8081 --root ./dist-staging --log ./logs/staging.log &

# Local preview of a feature branch
./PureSimpleHTTPServer --port 9000 --root ./feature-branch/dist &

Each process writes its own access log and maintains its own PID file (--pid-file), making it straightforward to manage them with a process supervisor such as systemd, supervisord, or launchd.

Log rotation with SIGHUP

PureSimpleHTTPServer reopens its log file when it receives SIGHUP. This integrates cleanly with logrotate: after rotating the file, send the signal and the server will start writing to the new path without dropping a single request. 

# /etc/logrotate.d/pshs
/var/log/pshs/access.log {
    daily
    rotate 14
    compress
    missingok
    notifempty
    postrotate
        kill -HUP $(cat /run/pshs.pid)
    endscript
}

Pass the PID file path to the server at startup with --pid-file /run/pshs.pid so the postrotate script can find it. The --log-keep and --log-size flags provide a simpler built-in rotation option if you prefer not to use logrotate.

Embedding assets at compile time

For truly single-file deployment — useful for distributing demo tools or internal utilities — you can embed your entire wwwroot into the binary at compile time using PureBasic's IncludeBinary and a small bundler script. The server then extracts the virtual file system into a temporary directory on first run and serves from there. 

# Bundle step (run once at build time)
./tools/bundle-wwwroot --input ./wwwroot --output ./src/EmbeddedAssets.pb

# Compile with embedded assets
pbcompiler PureSimpleHTTPServer.pb --embedded

The resulting binary carries every HTML, CSS, JS, and image file inside it. Users download one file, run it, and get the full experience — no separate asset archive required. This is how the demo site you are reading right now is distributed.

SPA mode for React, Vue, and Angular

Single-page applications manage routing entirely in the browser. When a user bookmarks /dashboard/settings and navigates directly to it, the server must return index.html rather than a 404. Enable SPA mode with one flag: 

./PureSimpleHTTPServer --port 3000 --root ./dist --spa

In SPA mode, any path that does not match a real file or a rewrite.conf rule falls through to index.html at the root. Static assets (.js, .css, .png, etc.) are still served normally — only unknown paths get the SPA fallback.

You can combine --spa with rewrite.conf rules; explicit rewrites are evaluated first. A common pattern is to use rewrites for versioned API paths and let SPA mode catch everything else: 

# wwwroot/rewrite.conf
rewrite /api/users    /api/v1/users.json
rewrite /api/products /api/v1/products.json
# All other unknown paths fall through to index.html (SPA mode)

Performance — what to expect

PureSimpleHTTPServer is built on PureBasic's native socket library with a multi-threaded connection handler. On a modern laptop (Apple M3, macOS Sequoia), a wrk load test against a small cached HTML file gives:

30,000 req/s

Peak throughput serving a 4 KB HTML file over localhost with 12 concurrent connections.

279 MB/s

Aggregate transfer rate on the same benchmark — effectively limited by the loopback interface.

< 3 ms

Median latency under load. The 99th-percentile stays below 8 ms on an unloaded machine.

For the vast majority of development and lightweight production workloads, raw throughput is not the bottleneck — it is network bandwidth and the size of your assets. Pre-compressing assets with gzip -k and serving .gz sidecars is the single most effective optimisation you can make.

Security — hidden paths and running behind nginx

PureSimpleHTTPServer automatically blocks requests that contain path traversal sequences (../) or that target hidden files and directories (names beginning with .). This means .git/, .env, and similar sensitive paths return 403 rather than leaking content.

For production HTTPS deployments, the recommended approach is to run PureSimpleHTTPServer on a non-public port (e.g., 127.0.0.1:8080) and place nginx in front of it to handle TLS termination: 

# nginx snippet — /etc/nginx/sites-available/mysite
server {
    listen 443 ssl http2;
    server_name mysite.example.com;

    ssl_certificate     /etc/letsencrypt/live/mysite.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/mysite.example.com/privkey.pem;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

This pattern gives you TLS, HTTP/2, and nginx's mature security hardening while keeping PureSimpleHTTPServer focused on what it does best: serving static files at high speed.
