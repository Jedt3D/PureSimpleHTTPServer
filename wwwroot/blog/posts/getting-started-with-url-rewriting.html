Getting Started with URL Rewriting

Published 2026-03-10

Clean URLs are one of the most visible quality signals of a well-crafted website. Visitors (and search engines) trust /about over /about.html, and /blog/hello-world over /blog/posts/hello-world.html. PureSimpleHTTPServer makes it straightforward to achieve both without touching your file structure — through a combination of the --clean-urls flag and plain-text rewrite.conf files.

What URL rewriting is and why it matters

URL rewriting is the practice of mapping one URL to another at the server level without redirecting the browser. When a visitor requests /about, the server quietly reads about.html and returns it — the address bar never changes.

There are three main reasons to care about it:

  • Clean URLs — Remove extensions and hierarchy noise so URLs are shorter, more memorable, and friendlier to share.
  • Permanent redirects — When you reorganise a site, 301 redirects preserve search-engine ranking and update stored bookmarks automatically.
  • Site migration — Map old URL patterns to a new structure in one place, so old links from other sites continue to work.

The quick win — --clean-urls

For the most common case — serving every .html file without its extension — you do not need a rewrite.conf at all. Just add --clean-urls: 

./PureSimpleHTTPServer --port 8080 --root ./wwwroot --clean-urls

With this flag enabled, the server automatically tries <path>.html whenever a request arrives for a URL that has no extension and does not match a directory. So /about, /docs/quickstart, and /blog/hello-world all resolve to their .html counterparts on disk with no configuration required.

--clean-urls handles the mechanical one-to-one mapping. For anything more structural — like mapping /blog/hello-world to /blog/posts/hello-world.html — you need a rewrite.conf.

Creating a rewrite.conf

A rewrite.conf is a plain-text file you place anywhere inside your wwwroot tree. PureSimpleHTTPServer reads the file closest to the requested URL first, then falls through to the parent directory's rules.

The file format supports three pattern types:

TypeSyntaxExample pattern
Exact rewrite /path /target rewrite /about /about.html
Glob rewrite /prefix/* /target/{path}.ext rewrite /blog/* /blog/posts/{path}.html
Regex regex rewrite ~/pattern /target/{re.1} regex rewrite ~/user/([0-9]+) /api/v1/users/{re.1}

The rewrite directive serves a different file internally — the browser never sees a redirect. The redir directive sends an HTTP 301 or 302, which tells the browser to navigate to the new URL.

Real example — blog slug routing

A typical blog keeps post files inside a posts/ subdirectory but wants to expose them at a cleaner top-level path. Place this in wwwroot/rewrite.conf: 

# wwwroot/rewrite.conf

# Map /blog/<slug> to the actual HTML file under posts/
rewrite /blog/* /blog/posts/{path}.html

This single rule maps:

  • /blog/hello-world → serves /blog/posts/hello-world.html
  • /blog/getting-started-with-url-rewriting → serves /blog/posts/getting-started-with-url-rewriting.html
  • /blog/advanced-tips → serves /blog/posts/advanced-tips.html

The * wildcard captures exactly one path segment. The captured value is substituted wherever {path} appears in the target. Two additional placeholders are available when the captured segment contains a slash: {dir} for the directory portion and {file} for the filename.

Per-directory rewrite.conf

Rather than putting all rules in a single root-level file, you can distribute rewrite.conf files throughout the tree. Each file governs only the directory it lives in and its subdirectories: 

wwwroot/
  rewrite.conf          ← global rules (applied last)
  blog/
    rewrite.conf        ← rules for /blog/* (applied first for blog URLs)
    posts/
      …
  api/
    rewrite.conf        ← rules for /api/* (applied first for API URLs)
    v1/
      …

Subdirectory rules always take precedence over the global rules, so you can override behaviour for a specific section without touching the root configuration. This demo site uses per-directory files in both blog/ and api/ — check them out as working examples.

Redirect status codes — 301 vs 302

When you use redir instead of rewrite, you must also provide an HTTP status code: 

# Permanent — search engines update their index, browsers cache the redirect
redir /old-page /new-page 301

# Temporary — browsers do not cache, search engines keep the original URL
redir /promo /sale-landing 302

The rule of thumb: use 301 for anything you have moved permanently (renamed pages, reorganised site structure, domain migrations). Use 302 for temporary promotions, A/B test variants, or maintenance pages that will disappear in the future.

If you are unsure, lean towards 302 during development — you can always promote it to 301 once you are confident the new URL is stable. Changing a 301 back is harder because browsers may have cached it aggressively.
